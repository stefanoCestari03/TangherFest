1. Contesto e requisiti del progetto

Il sito TangherFest Green Volley 3×3 è un'applicazione React/Vite che gestisce le iscrizioni al torneo di beach volley. Il database deve supportare:

  • Iscrizione squadre con dati di tutti i giocatori (3 obbligatori + 1 facoltativo)
  • Upload documenti: documento di identità di ogni giocatore, eventuale tessera FIPaV, documento del tutore per i minorenni
  • Consensi digitali con valore legale: timestamp, IP, versione documento, user-agent
  • Aggiornamenti in tempo reale: la pagina Squadre si aggiorna automaticamente quando arriva una nuova iscrizione
  • Limite posti: 12 squadre tesseratem + 12 libere (max 24 totali)
  • Accesso in sola lettura pubblico alla lista squadre; scrittura solo tramite form di iscrizione

Volumi attesi

DatoStima
Squadre totaliMax 24
Giocatori totaliMax ~96 (24 × 4)
Documenti caricati (file)Max ~200 file, ~500MB totali
Letture giornaliere (utenti che visitano la pagina)~500–2.000
Scritture (iscrizioni)24 in tutto il periodo
Connessioni realtime simultanee~50–100 durante picco
Nota: Per questi volumi qualsiasi soluzione modern-cloud va bene. La scelta va guidata da funzionalità integrate, non da scalabilità.

3. Perché Supabase — motivazione dettagliata

Supabase è la scelta ottimale per questo progetto per 6 ragioni concrete:

  1. Già integrato nel codice — src/lib/supabase.js, src/lib/db.js e la subscription realtime sono già scritti. Attivarlo richiede solo le variabili d'ambiente.
  2. PostgreSQL = dati strutturati — Le iscrizioni al torneo sono dati fortemente strutturati (non documenti JSON arbitrari). PostgreSQL garantisce integrità referenziale, constraints, e query SQL potenti per reportistica.
  3. Storage integrato — I documenti di identità e le tessere FIPaV vengono caricati direttamente nel bucket volley-docs di Supabase. Nessun servizio esterno (AWS S3, Cloudinary, ecc.).
  4. Realtime out-of-the-box — La pagina Squadre si aggiorna in tempo reale via WebSocket quando arriva una nuova iscrizione, senza polling. Già implementato in subscribeSquadre().
  5. Row Level Security (RLS) — Regole di sicurezza a livello di riga direttamente nel DB. Lettura pubblica delle squadre (senza autenticazione), scrittura solo tramite la chiave anonima del form. Nessuna API custom da scrivere.
  6. Free tier sufficiente — Per 24 squadre e un evento annuale, il tier gratuito di Supabase (500MB DB, 1GB storage, 50.000 richieste/mese) è abbondantemente sufficiente per anni.
Alternativa valida se già si usa Firebase: Firebase Firestore funzionerebbe bene per i dati e Firebase Storage per i documenti. Il realtime è nativo. Lo svantaggio è che i dati NoSQL rendono più difficile fare query come "tutti i giocatori di categoria Serie A" o creare report in Excel. Richiederebbe anche riscrivere tutto il layer db.js.

6. SQL Scripts completi

Script 1 — Creazione tabella (Opzione A — JSONB)

Copia e incolla questo script nel SQL Editor di Supabase e premi Run.

-- ════════════════════════════════════════════════════
-- TangherFest Green Volley 3×3 — Schema database v1.0
-- Opzione A: JSONB (consigliata per il lancio)
-- ════════════════════════════════════════════════════

-- Tabella principale: una riga = una squadra iscritta
CREATE TABLE IF NOT EXISTS squadre (
  id                  TEXT PRIMARY KEY,
  "nomeSquadra"       TEXT NOT NULL,
  referente           TEXT NOT NULL,
  email               TEXT NOT NULL,
  telefono            TEXT NOT NULL,
  tipo                TEXT NOT NULL CHECK (tipo IN ('tesserata', 'libera')),
  giocatori           JSONB NOT NULL DEFAULT '[]',
  metadati            JSONB NOT NULL DEFAULT '{}',
  creato_il           TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indici per velocizzare le query più comuni
CREATE UNIQUE INDEX IF NOT EXISTS idx_squadre_email
  ON squadre (LOWER(email));

CREATE UNIQUE INDEX IF NOT EXISTS idx_squadre_nome
  ON squadre (LOWER("nomeSquadra"));

-- Indice GIN per query sui giocatori (es. cerca per CF)
-- Utile se in futuro vuoi query come:
-- WHERE giocatori @> '[{"codiceFiscale": "RSSMRC95C15A562Z"}]'
CREATE INDEX IF NOT EXISTS idx_squadre_giocatori
  ON squadre USING GIN (giocatori);

-- View utile per contare i posti disponibili
CREATE OR REPLACE VIEW contatori_iscrizioni AS
SELECT
  COUNT(*) FILTER (WHERE tipo = 'tesserata') AS n_tesseratem,
  COUNT(*) FILTER (WHERE tipo = 'libera')    AS n_libere,
  COUNT(*)                                     AS totale
FROM squadre;

Script 2 — Row Level Security (RLS)

-- ════════════════════════════════════════════════════
-- Row Level Security — accesso controllato per ruolo
-- ════════════════════════════════════════════════════

-- Attiva RLS sulla tabella (obbligatorio per sicurezza)
ALTER TABLE squadre ENABLE ROW LEVEL SECURITY;

-- Policy 1: lettura pubblica
-- Chiunque può leggere le squadre iscritte (pagina Squadre del sito)
CREATE POLICY "Lettura pubblica squadre"
  ON squadre
  FOR SELECT
  TO anon, authenticated
  USING (true);

-- Policy 2: inserimento tramite chiave anonima
-- Il form di iscrizione usa la chiave "anon" per inviare nuove iscrizioni
CREATE POLICY "Inserimento iscrizioni"
  ON squadre
  FOR INSERT
  TO anon
  WITH CHECK (true);

-- NOTA: Nessuna policy di UPDATE o DELETE per utenti anonimi.
-- Le modifiche sono possibili solo dal pannello Supabase (service role).
-- Se in futuro aggiungi un pannello admin, crea una policy
-- per il ruolo "authenticated" con verifica email.

Script 3 — Storage bucket

-- ════════════════════════════════════════════════════
-- Crea il bucket per i documenti caricati dagli utenti
-- ════════════════════════════════════════════════════

-- Crea il bucket (eseguire UNA SOLA VOLTA)
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'volley-docs',
  'volley-docs',
  false,                          -- privato: i file non sono accessibili via URL pubblico
  5242880,                        -- max 5MB per file
  ARRAY['image/jpeg', 'image/png', 'application/pdf']
);

-- Policy storage: solo upload anonimo (form di iscrizione)
CREATE POLICY "Upload documenti iscrizione"
  ON storage.objects
  FOR INSERT
  TO anon
  WITH CHECK (bucket_id = 'volley-docs');

-- Policy storage: lettura riservata agli admin (authenticated)
-- Il pubblico NON può scaricare i documenti di identità altrui
CREATE POLICY "Lettura documenti solo admin"
  ON storage.objects
  FOR SELECT
  TO authenticated
  USING (bucket_id = 'volley-docs');

7. Row Level Security — spiegazione approfondita

L'RLS (Row Level Security) di PostgreSQL permette di definire chi può leggere e scrivere ogni riga direttamente nel database, senza dipendere dalla logica dell'applicazione.

Architettura di accesso per questo progetto

OperazioneChiComePolicy
Leggere le squadre iscritte Qualsiasi visitatore chiave anon SELECT → USING (true)
Iscrivere una squadra Form di iscrizione (frontend) chiave anon INSERT → WITH CHECK (true)
Modificare/eliminare un'iscrizione Solo amministratori Pannello Supabase (service role) Nessuna policy pubblica
Scaricare i documenti Solo amministratori Pannello Supabase o API autenticata Storage: solo authenticated
Importante: Senza abilitare RLS (ENABLE ROW LEVEL SECURITY), la tabella è completamente aperta. Con RLS attivo ma senza policy, la tabella è completamente chiusa. Le policy vanno create dopo aver abilitato RLS.

8. Configurazione Storage

Il bucket volley-docs conterrà tutti i documenti caricati dai partecipanti. La struttura dei path è:

volley-docs/
  docs/
    {id-squadra}_g1_{nome-file}            ← documento giocatore 1
    {id-squadra}_g2_{nome-file}            ← documento giocatore 2
    {id-squadra}_g3_{nome-file}            ← documento giocatore 3
    {id-squadra}_g4_{nome-file}            ← documento giocatore 4 (se presente)
    {id-squadra}_g1_tutore_{nome-file}     ← documento tutore giocatore minorenne
    {id-squadra}_g2_tutore_{nome-file}     ← ecc.

Accesso ai documenti dal pannello Supabase

1
Vai su Storage nel menu laterale di Supabase.
2
Clicca sul bucket volley-docs → cartella docs.
3
Puoi scaricare ogni file cliccandoci sopra → "Download".
Il bucket è privato (public: false). Per generare un link temporaneo condivisibile usa l'API supabase.storage.from('volley-docs').createSignedUrl(path, 3600) dalla console Supabase o da uno script admin.

10. Realtime — come funziona

Quando un utente è sulla pagina Squadre e qualcun altro invia un'iscrizione, la pagina si aggiorna automaticamente senza ricaricare. Ecco il flusso:

Utente A: compila il form e invia
    │
    ▼
Supabase: INSERT INTO squadre (...)
    │
    ▼
Postgres notifica il canale "public:squadre"
    │
    ▼
Supabase Realtime WebSocket → broadcast a tutti i client connessi
    │
    ▼
App.jsx: onInsert(nuovaSquadra) → setSquadre(prev => [...prev, nuova])
    │
    ▼
React re-render → SquadrePage mostra la nuova squadra in tempo reale

La funzione subscribeSquadre in src/lib/db.js gestisce tutto questo. La subscription viene aperta al mount dell'app e chiusa all'unmount (cleanup automatico con return unsubscribe in useEffect).

Prerequisito realtime: La tabella deve essere aggiunta alla pubblicazione realtime di Supabase. Lo fa lo Script 4 di questo documento (ALTER PUBLICATION supabase_realtime ADD TABLE squadre).

12. Modificare le tabelle dopo la creazione

Sì, puoi modificarle. PostgreSQL (e quindi Supabase) permette di alterare le tabelle in qualsiasi momento con il comando ALTER TABLE, eseguibile dal SQL Editor di Supabase senza dover ricreare nulla da zero.

Come accedere: Supabase → SQL Editor → New query → incolla il comando → Run.

Operazioni più comuni

Aggiungere una colonna

-- Aggiunge una colonna opzionale alla tabella squadre
ALTER TABLE squadre
  ADD COLUMN note TEXT;

-- Con valore di default per le righe esistenti
ALTER TABLE squadre
  ADD COLUMN confermata BOOLEAN NOT NULL DEFAULT FALSE;

Rinominare una colonna

ALTER TABLE squadre
  RENAME COLUMN "nomeSquadra" TO nome_squadra;

-- ATTENZIONE: dopo il rename devi aggiornare anche il codice JS
-- in src/lib/db.js e src/components/ ovunque si usa il vecchio nome

Rimuovere una colonna

-- Elimina definitivamente la colonna e tutti i suoi dati
ALTER TABLE squadre
  DROP COLUMN IF EXISTS note;

Modificare il tipo di una colonna

-- Cambia il tipo (solo se i dati esistenti sono compatibili)
ALTER TABLE squadre
  ALTER COLUMN telefono TYPE VARCHAR(20);

Aggiungere un constraint (unicità, check)

-- Aggiunge vincolo unicità sul nome squadra (case-insensitive)
CREATE UNIQUE INDEX IF NOT EXISTS idx_squadre_nome
  ON squadre (LOWER("nomeSquadra"));

-- Aggiunge check sul tipo
ALTER TABLE squadre
  ADD CONSTRAINT chk_tipo CHECK (tipo IN ('tesserata', 'libera'));

Aggiungere un indice per velocizzare le query

-- Indice sul campo tipo (utile per la query "conta le tesseratem")
CREATE INDEX IF NOT EXISTS idx_squadre_tipo ON squadre (tipo);

-- Indice GIN su JSONB (per query dentro l'array giocatori)
CREATE INDEX IF NOT EXISTS idx_squadre_giocatori
  ON squadre USING GIN (giocatori);
